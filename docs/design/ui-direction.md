# 正式 UI 与视觉方向

## 世界观提炼

### 核心命题

2038 年，AI 接管大部分工作。玩家不是拯救世界的英雄，而是在“大淘汰时代”中重新选择职业、积累能力、押注风口的普通人。

### 核心情绪曲线

```text
焦虑 → 挣扎 → 发现机会 → 押注未来 → 逆袭或承担失败
```

正式 UI 不能只表达压抑。它需要同时呈现：

- 岗位不断减少的紧迫感；
- 选择职业赛道的策略感；
- 能力持续升级的成长感；
- 押中风口时的兴奋；
- 结算时具有黑色幽默的个人命运故事。

### 机制视觉映射

| 机制 | 世界观表达 | UI 表达建议 |
|---|---|---|
| 牌堆 | 尚未开放的岗位池 | 岗位档案、职位队列、剩余名额 |
| 手牌 | 当前掌握的选择 | 私人履历夹、机会卡 |
| 五色列 | 五条职业赛道 | 职业发展轨迹、能力阶梯 |
| 数字牌 | 能力等级 | 职级、技能认证、成长节点 |
| 加成牌 | 风口 | 趋势报告、早期投资标记 |
| 万能牌 | 贵人或奇迹 | 特殊推荐、偶然突破 |
| 弃牌堆 | 人才市场 | 公开岗位市场、被放弃的机会 |
| 20 分门槛 | 转型成本 | 学费、时间、风险、试错投入 |
| 倒计时 | 操作窗口 | 招聘窗口关闭倒计时 |
| 最后一张牌 | 最后岗位被领取 | 全局机会窗口关闭 |

## 视觉方向候选

## A. 近未来人才交易所

### 核心气质

理性、克制、紧张，像 2038 年的公共就业终端与人才交易大厅。不是炫技赛博朋克，而是“看起来真实得令人不安”的近未来。

### 色彩方向

- 主背景：石墨黑、深蓝灰。
- 信息层：冷白、低饱和蓝。
- 警告与倒计时：橙红。
- 五赛道：保留红、蓝、黄、绿、白，但降低纯度并建立无障碍辅助图标。
- 成功状态：偏青绿，不使用廉价霓虹绿。

### 字体感觉

- 标题：窄体、略具工业感的无衬线字体。
- 正文与数字：清晰的现代无衬线字体。
- 倒计时、职级和岗位编号：可使用等宽数字。
- 中文字体应优先保证移动端可读性，不建议大面积使用装饰字体。

### 卡牌视觉

- 像岗位档案、技能认证和趋势简报。
- 左上角显示赛道图标与颜色，中央突出能力等级。
- 风口牌采用“趋势预测/投资评级”语言。
- 贵人牌使用特殊推荐章、金属箔或异常权限标识。
- 卡牌必须同时使用颜色、图标和文字，不能只靠颜色区分。

### 房间大厅

包装成“转型评估中心”或“候选人登记处”。玩家列表像排队中的候选档案，房主是本次评估发起人。

### 游戏桌面

像多人共享的职业规划终端。中央是岗位池和人才市场，玩家面前是五条横向或纵向能力轨迹。

### MVP 适配

非常适合。现有信息密度较高，终端式视觉可以自然容纳房间号、连接状态、倒计时和日志。

### 实现难度

中等。主要工作是设计系统、卡牌组件、图标和响应式布局，不依赖大量插画。

### 推荐程度

**5/5，首选方向。**

---

## B. 黑色职场寓言

### 核心气质

冷峻、讽刺、压迫，接近企业年报、绩效面板和自动化裁员通知构成的黑色幽默。

### 色彩方向

- 黑、灰白、暗红为主。
- 五赛道只在卡牌边缘和数据图表中出现。
- 大面积留白配合强警告色，营造“正式文件比怪物更可怕”的感觉。

### 字体感觉

- 标题接近报刊或正式公文的粗衬线体。
- 数据区域使用企业系统常见无衬线体。
- 印章、批注和删除线可作为少量装饰。

### 卡牌视觉

- 像绩效报告、劳动合同、解雇通知和投资备忘录。
- 数字牌是职级晋升记录。
- 风口牌是董事会预测或资本评级。
- 贵人牌是内部推荐、破格录用或偶然爆款。

### 房间大厅

像等待面试或裁员复核的会议室。玩家进入后显示候选编号、在线状态和本局规则。

### 游戏桌面

像一张被多人批注的职业档案桌，日志可包装成自动决策记录。

### MVP 适配

适合，但需要谨慎控制压抑程度。若文案全是负面通知，长期游玩会疲劳。

### 实现难度

中等偏高。高度依赖文案、排版细节和统一的讽刺语气，做不好容易像普通后台管理系统。

### 推荐程度

**4/5，适合作为首选方向的叙事层。**

---

## C. 乐观复古未来主义

### 核心气质

明亮、快速、有希望，采用过去想象未来时的宣传画、职业手册和模块化信息图风格。重点是“重新发明自己”，而不是“等待被淘汰”。

### 色彩方向

- 米白或浅灰作为底色。
- 五赛道使用更活跃但不过分儿童化的颜色。
- 深蓝负责结构，橙色负责机会和行动。
- 少量网点、套色偏移和纸张纹理增强印刷感。

### 字体感觉

- 标题使用几何粗体。
- 正文使用友好的现代无衬线体。
- 数字等级要大而有节奏，接近交通导视或旧式职业海报。

### 卡牌视觉

- 像未来职业招募海报。
- 每条赛道拥有抽象图形语言，不必立即制作大量人物插画。
- 风口牌可以做成夸张的趋势广告。
- 贵人牌采用高对比的“特殊机会”构图。

### 房间大厅

像职业博览会入口。玩家是即将进入转型实验的参与者。

### 游戏桌面

像色彩鲜明的实体桌游桌面，卡牌和赛道更容易快速识别。

### MVP 适配

适合移动端和大众传播，但需要避免过度轻松，削弱故事中的焦虑与代价。

### 实现难度

中等偏高。需要较成熟的图形系统、纹理和插画方向。

### 推荐程度

**4/5，适合更大众化的产品定位。**

## 推荐视觉方向

推荐以 **A「近未来人才交易所」为结构基础**，吸收 B 的黑色幽默文案，并少量采用 C 的成长反馈。

具体比例建议：

```text
70% 近未来人才终端
20% 黑色职场寓言
10% 逆袭成功时的明亮反馈
```

这样既能承载当前复杂联机信息，也不会把产品做成传统后台或纯粹压抑的反乌托邦。

### 建议设计关键词

```text
职业档案 / 岗位窗口 / 能力轨迹 / 趋势报告
克制未来 / 清晰数据 / 有限机会 / 黑色幽默
```

### 不建议

- 不建议使用城堡、王冠、盾徽和羊皮纸。
- 不建议使用满屏霓虹、故障字和廉价赛博朋克素材。
- 不建议把五色直接做成没有图标辅助的纯色块。
- 不建议大量使用真实公司或真实职业群体的负面梗。

## 正式 UI 信息架构

## 1. 首页

目标是让新玩家在十秒内理解“这是什么、如何开始”。

### 内容

- 正式游戏标题。
- 一句 slogan，例如：“最后一个岗位被拿走前，完成你的转型。”
- 2–4 人多人策略卡牌游戏说明。
- 昵称输入。
- 主操作：创建房间。
- 次操作：加入房间。
- 房间号输入可放在加入面板内，避免首页同时出现过多输入框。
- 规则入口。
- 当前连接状态。
- 版本号与 Debug 入口仅在开发环境显示。

### 状态

- 连接中：主按钮暂不可用，并明确说明正在连接。
- 连接失败：提供重试，不清空有效本地会话。
- 检测到本地会话：优先显示“正在恢复上局”。

## 2. 创建房间面板

### 内容

- 昵称。
- 玩家人数：建议首版显示 `2–4 人`，但不承诺固定人数锁房，除非服务端增加该配置。
- 限时模式：
  - 无限时；
  - 标准：20 秒基础时间 + 50 秒额外时间；
  - 宽松：30 秒基础时间 + 80 秒额外时间。
- 每种模式的一句简短说明。
- 创建房间按钮。

### 交互

- 默认选择标准模式。
- 记住玩家最近使用的昵称和模式，但不要覆盖有效游戏会话。
- 创建期间显示 pending 状态，防止重复提交。

## 3. 房间等待页

### 内容

- 房间号，使用醒目的短码区域。
- “复制邀请信息”按钮，文案包含游戏链接和房间号。
- 玩家列表、房主标记、在线/离线状态。
- 当前人数与支持人数。
- 本局限时模式。
- 房主开始游戏按钮。
- 非房主显示“等待房主开始”。
- 离开房间入口。

### 反馈

- 新玩家加入时有轻量提示。
- 不使用只靠颜色表达在线状态。
- 人数不足时解释为什么不能开始。

## 4. 游戏桌面页

### 桌面层级

1. **全局状态栏**
   - 房间号；
   - 当前玩家；
   - 当前阶段；
   - 倒计时；
   - 当前玩家额外时间；
   - 连接/重连状态。

2. **其他玩家区域**
   - 昵称和在线状态；
   - 手牌数量；
   - 五条公开职业轨迹；
   - 当前玩家高亮；
   - 不显示其他玩家具体手牌。

3. **公共机会区**
   - 剩余岗位数量；
   - 从岗位池摸牌；
   - 五个人才市场弃牌堆；
   - 顶牌突出；
   - 可展开查看每堆完整历史和顺序。

4. **自己的职业区域**
   - 五条职业赛道；
   - 风口牌位于轨迹起点；
   - 能力数字形成清晰递增路径；
   - 万能牌明确标注其永久角色，避免只显示“万能”。

5. **自己的手牌**
   - 固定在桌面底部；
   - 支持选择、取消和查看详情；
   - 可放置目标给出合法性提示；
   - 出牌阶段突出放置/弃置，摸牌阶段隐藏无关操作。

6. **辅助区域**
   - 事件日志；
   - 规则提示；
   - 网络错误和非法动作提示；
   - 最终排名与每条赛道计分明细。

### 桌面端布局

```text
┌──────────────── 当前玩家 / 阶段 / Timer / 连接状态 ────────────────┐
│                    其他玩家职业轨迹区域                            │
├───────────────┬─────────────────────────────┬─────────────────────┤
│ 事件日志/规则  │      岗位池 + 五个人才市场     │ 玩家与房间摘要       │
├───────────────┴─────────────────────────────┴─────────────────────┤
│                     自己的五条职业赛道                            │
├───────────────────────────────────────────────────────────────────┤
│                         自己的手牌                                │
└───────────────────────────────────────────────────────────────────┘
```

### 移动端布局

- 顶部固定显示当前阶段和倒计时。
- 其他玩家折叠为横向卡片。
- 公共机会区保持可见。
- 自己的五条赛道使用横向滚动或紧凑分组。
- 手牌固定在底部，可横向滚动。
- 日志、规则和完整弃牌历史进入抽屉。
- 主要操作触控区域至少约 44px，不依赖 hover。

## 5. 最终排名

- 明确展示并列冠军和竞赛排名。
- 显示总分和五条赛道得分。
- 根据玩家轨迹生成一至两句赛后命运摘要，例如：
  - “押中科技风口，但转型成本仍未收回。”
  - “没有追逐风口，却在医疗赛道稳步成为行业领袖。”
- 提供“再来一局”和“返回房间”前，应先确认服务端是否支持对应流程。

## Debug 页面建议

建议保留当前调试 UI，并迁移为：

```text
/debug
```

### 原因

- 可以直接查看 room、player、phase、timer 和事件类型，便于排查联机问题。
- 正式 UI 会隐藏大量技术细节，Debug 页面能缩短线上试玩反馈的定位时间。
- 可作为 Playwright 的稳定测试页面，降低视觉重构导致测试大面积失效的风险。
- 新规则或 Socket 协议可先在 Debug 页面验证，再接入正式表现层。

### 约束

- 生产环境默认不显示入口，可通过明确 URL 或环境变量开启。
- Debug 页面仍必须使用 `PlayerGameView`，不能暴露完整 `GameState` 或牌堆顺序。
- 不应在 Debug 页面展示或复制 `sessionToken`。

## 组件拆分建议

建议目标结构：

```text
src/client/
  pages/
    HomePage.tsx
    RoomLobbyPage.tsx
    GamePage.tsx
    DebugPage.tsx
  components/
    ConnectionStatus.tsx
    CreateRoomPanel.tsx
    JoinRoomPanel.tsx
    RoomLobby.tsx
    GameBoard.tsx
    Hand.tsx
    GameCard.tsx
    PlayerArea.tsx
    CareerColumn.tsx
    DiscardPiles.tsx
    TimerPanel.tsx
    EventLog.tsx
    FinalRanking.tsx
    ErrorBanner.tsx
  hooks/
    useGameSocket.ts
    useSavedSession.ts
    useServerTimer.ts
```

为避免与 DOM 原生 `Card`、领域类型 `Card` 混淆，React 卡牌组件建议命名为 `GameCard.tsx`。

### 第一优先级

1. `useGameSocket`
   - 统一监听器、ack、pending、错误和自动重连。
   - 让页面组件不直接操作 Socket.IO 单例。

2. `ConnectionStatus`、`TimerPanel`
   - 状态边界清楚，且正式视觉高度依赖这两个组件。

3. `Hand`、`GameCard`、`CareerColumn`
   - 是正式卡牌视觉系统的基础。
   - 需要集中处理选中、禁用、合法目标和万能牌角色。

4. `GameBoard`、`PlayerArea`、`DiscardPiles`
   - 决定桌面布局和响应式策略。

5. `HomePage`、`CreateRoomPanel`、`JoinRoomPanel`、`RoomLobby`
   - 形成正式入口和等待流程。

### 第二优先级

- `EventLog`
- `FinalRanking`
- `ErrorBanner`
- 规则抽屉
- 完整人才市场历史面板

这些组件重要，但可以在核心桌面布局稳定后再做精细视觉。

### 可以暂时不拆

- 简单颜色/赛道标签。
- 小型格式化函数。
- 仅一两行内容的状态提示。

不要为了文件数量而拆分无状态薄组件。

### 与正式 UI 强相关

- `GameCard`
- `CareerColumn`
- `Hand`
- `GameBoard`
- `PlayerArea`
- `DiscardPiles`
- `TimerPanel`
- `FinalRanking`

这些组件应在视觉令牌、卡牌尺寸、响应式断点和交互状态确定后实现。

### 应保持 Debug 风格

- 原始事件日志。
- roomId/playerId 技术信息。
- Socket 连接细节。
- 原始阶段名和事件类型。
- 用于测试的 data-testid 与状态快照。

Debug 表现不应污染正式页面，但底层数据来源可以共用。

## 下一步实现计划

### 阶段 1：命名与视觉验证

1. 对 Top 3 名称进行商标、商店、域名和同类产品查重。
2. 用 5–8 名目标玩家做名称与 slogan 快速访谈。
3. 为推荐视觉方向制作首页、卡牌、游戏桌面三张静态 moodboard。
4. 确认五条职业赛道的正式名称、图标和无障碍颜色。

### 阶段 2：设计系统

1. 定义颜色、字体、间距、圆角、阴影和动效令牌。
2. 定义卡牌的普通、选中、可放置、不可放置、风口、贵人状态。
3. 定义连接、倒计时、额外时间和错误状态。
4. 完成桌面端与移动端低保真布局。

### 阶段 3：技术拆分

1. 保留当前 UI 为 `/debug`。
2. 抽取 Socket/session/timer hooks。
3. 拆分纯展示组件，不改变规则引擎和协议。
4. 为关键组件建立独立测试或 Storybook 类展示环境；是否引入新工具另行评估。

### 阶段 4：正式 UI 落地

1. 首页与房间大厅。
2. 卡牌和手牌。
3. 五赛道游戏桌面。
4. 人才市场、Timer 和其他玩家区域。
5. 日志、错误提示和最终排名。
6. 移动端适配与真实设备试玩。

### 阶段 5：叙事增强

1. 为牌面编写正式名称和短文案。
2. 为风口牌与贵人牌建立内容库。
3. 根据最终轨迹生成赛后职业故事。
4. 在不泄露隐藏信息的前提下增强事件日志的叙事表达。
